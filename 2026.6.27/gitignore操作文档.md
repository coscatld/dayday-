# .gitignore 语法与操作文档

## 1. .gitignore 是什么

`.gitignore` 是 Git 的忽略规则文件，用来告诉 Git 哪些文件或目录不需要纳入版本管理。

常见需要忽略的内容包括：

- Python 缓存文件，例如 `__pycache__/`、`*.pyc`
- 虚拟环境目录，例如 `.venv/`、`venv/`
- 构建输出目录，例如 `build/`、`dist/`
- IDE 配置目录，例如 `.idea/`、`.vscode/`
- 本地数据库、日志、临时文件，例如 `*.db`、`*.log`、`*.tmp`
- 本地环境变量文件，例如 `.env`

注意：`.gitignore` 只会影响“还没有被 Git 跟踪”的文件。如果某个文件已经被 `git add` 或提交过，后来再写入 `.gitignore`，Git 仍然会继续跟踪它。

## 2. 基本语法

`.gitignore` 是一行一个规则。

示例：

```gitignore
# Python 缓存
__pycache__/
*.pyc

# 打包输出
build/
dist/

# 本地配置
.env
.idea/
```

## 3. 注释

以 `#` 开头的行是注释，Git 不会把它当作忽略规则。

```gitignore
# 这是注释
```

如果文件名本身就是以 `#` 开头，需要用反斜杠转义：

```gitignore
\#example.txt
```

## 4. 忽略文件

直接写文件名，表示忽略这个文件名。

```gitignore
fris_data.db
.env
```

这会忽略仓库中任意位置叫这些名字的文件。

## 5. 忽略某类文件

使用 `*` 表示通配。

```gitignore
*.log
*.tmp
*.pyc
```

含义：

- `*.log`：忽略所有 `.log` 文件
- `*.tmp`：忽略所有 `.tmp` 文件
- `*.pyc`：忽略所有 Python 编译缓存文件

## 6. 忽略目录

目录规则建议以 `/` 结尾。

```gitignore
build/
dist/
__pycache__/
.idea/
```

这表示忽略整个目录及其内部所有文件。

## 7. 根目录匹配

规则前面加 `/`，表示只匹配仓库根目录。

```gitignore
/fris_data.db
/dist/
```

区别：

```gitignore
fris_data.db
```

会忽略任意位置的 `fris_data.db`。

```gitignore
/fris_data.db
```

只忽略仓库根目录下的 `fris_data.db`。

## 8. 多级目录匹配

使用 `**` 可以匹配多级目录。

```gitignore
**/__pycache__/
logs/**/*.log
```

含义：

- `**/__pycache__/`：忽略所有层级中的 `__pycache__` 目录
- `logs/**/*.log`：忽略 `logs` 目录下任意层级的 `.log` 文件

## 9. 反向保留

使用 `!` 可以取消忽略，也就是“保留”某些文件。

```gitignore
*.log
!important.log
```

含义是：忽略所有 `.log` 文件，但保留 `important.log`。

常见例子：

```gitignore
*.xlsx
!煤矿财务分析系统_模板.xlsx
```

含义是：忽略所有 Excel 文件，但保留指定模板文件。

注意：如果父目录已经被忽略，直接保留子文件可能不会生效，需要同时保留父目录。

```gitignore
data/*
!data/
!data/example.xlsx
```

## 10. 规则顺序

`.gitignore` 规则从上到下读取，后面的规则可以覆盖前面的规则。

```gitignore
*.log
!debug.log
```

这里 `debug.log` 会被保留。

如果顺序反过来：

```gitignore
!debug.log
*.log
```

那么 `debug.log` 仍然会被忽略，因为后面的 `*.log` 覆盖了前面的保留规则。

## 12. PyInstaller 项目常见忽略项

如果项目使用 PyInstaller 打包，例如生成 `exe`，通常需要忽略：

```gitignore
build/
dist/
*.manifest
*.toc
*.pyz
*.pkg
*.spec.bak
```

说明：

- `build/`：PyInstaller 构建中间目录
- `dist/`：最终打包输出目录
- `*.toc`、`*.pyz`、`*.pkg`：打包过程中产生的中间文件
- `*.spec.bak`：备份的 spec 文件

通常 `.spec` 文件本身可以保留，因为它记录了打包配置。

## 14. 检查忽略规则是否生效

查看当前 Git 状态：

```bash
git status --short
```

查看被忽略的文件：

```bash
git status --short --ignored
```

检查某个文件为什么被忽略：

```bash
git check-ignore -v 文件路径
```

例如：

```bash
git check-ignore -v dist/app.exe
git check-ignore -v fris_data.db
```

如果命令输出了规则来源，说明该文件确实被 `.gitignore` 忽略了。

## 15. 已经被 Git 跟踪的文件怎么办

如果文件已经被 Git 跟踪，写进 `.gitignore` 后不会自动取消跟踪。

取消跟踪单个文件：

```bash
git rm --cached 文件名
```

例如：

```bash
git rm --cached fris_data.db
```

取消跟踪整个目录：

```bash
git rm -r --cached 目录名
```

例如：

```bash
git rm -r --cached .idea
```

注意：`--cached` 表示只从 Git 索引中移除，不删除本地文件。

执行后再提交：

```bash
git add .gitignore
git commit -m "Update gitignore rules"
```

## 16. 常见问题

### 16.1 为什么写了 .gitignore 但没有生效

常见原因：

- 文件已经被 Git 跟踪过
- 规则写错了
- `.gitignore` 不在仓库根目录
- 父目录被忽略后，子文件的 `!` 保留规则没有同时保留父目录

可以使用下面命令排查：

```bash
git check-ignore -v 文件路径
git status --short --ignored
```

### 16.2 .gitignore 应该提交到仓库吗

应该提交。

`.gitignore` 是团队协作规则的一部分，建议和源码一起提交。

### 16.3 数据库文件要不要提交

一般不建议提交运行时数据库，例如：

```gitignore
fris_data.db
*.sqlite
*.sqlite3
```

如果数据库是初始化模板或演示数据，可以考虑改名，例如：

```text
sample_data.db
demo_data.db
```

并在文档里说明它的用途。

### 16.4 Excel 模板要不要忽略

如果 Excel 文件是用户导入导出的临时数据，建议忽略。

如果 Excel 文件是系统运行需要的模板，建议保留。

例如：

```gitignore
*.xlsx
!煤矿财务分析系统_模板.xlsx
```

## 17. 推荐操作流程

新项目中使用 `.gitignore` 的推荐流程：

1. 创建 `.gitignore`
2. 写入适合项目的忽略规则
3. 执行 `git status --short --ignored` 检查结果
4. 如果发现已跟踪的无用文件，使用 `git rm --cached` 取消跟踪
5. 提交 `.gitignore`

示例：

```bash
git status --short --ignored
git rm -r --cached .idea
git rm --cached fris_data.db
git add .gitignore
git commit -m "Add gitignore"
```

## 18. 小结

记住几个最常用的规则就够用了：

```gitignore
# 忽略目录
dist/

# 忽略某类文件
*.log

# 只忽略根目录文件
/fris_data.db

# 反向保留
!important.log
```

`.gitignore` 的重点不是写得越多越好，而是把“源码和必要配置”留下，把“缓存、构建产物、本地环境、运行时数据”排除出去。
